Integrating Google Books API into a Django App

Kapil 2024 8 min read

NovelNet is a book discovery and review platform I built with Django. The core challenge was data: a good book platform needs a rich catalog of books with covers, descriptions, authors, and ISBNs. Building and maintaining that manually would be impossible. The Google Books API made it possible to have millions of books available from day one.

Why Google Books API?

Google Books has an enormous catalog — over 40 million books — with structured metadata including titles, authors, descriptions, cover images, categories, and publication dates. It's free for reasonable usage and well-documented. The main limitation is rate limits (1000 requests per day without an API key, much more with one), which we handled with caching.

The Django Model Design

I didn't want to store all book data in my database — that would be massive and require constant syncing. Instead, I used a hybrid approach: store only the books that users interact with (review, save, or search), and fetch the rest from the API on demand.

class Book(models.Model):
    google_books_id = models.CharField(max_length=100, unique=True)
    title = models.CharField(max_length=500)
    authors = models.JSONField(default=list)
    description = models.TextField(blank=True)
    thumbnail = models.URLField(blank=True)
    published_date = models.CharField(max_length=20, blank=True)
    categories = models.JSONField(default=list)
    cached_at = models.DateTimeField(auto_now=True)

    class Meta:
        indexes = [models.Index(fields=["google_books_id"])]

The google_books_id field is the key — it's Google's unique identifier for each book, and it's what we use to sync local records with the API.

The API Integration Layer

I wrote a service module to handle all Google Books API communication. Keeping API calls in a dedicated service layer (not in views, not in models) made it easy to mock in tests and swap out later.

import requests
from django.conf import settings

BOOKS_API_BASE = "https://www.googleapis.com/books/v1"

def search_books(query: str, max_results: int = 10) -> list[dict]:
    response = requests.get(
        f"{BOOKS_API_BASE}/volumes",
        params={
            "q": query,
            "maxResults": max_results,
            "key": settings.GOOGLE_BOOKS_API_KEY,
        },
        timeout=5,
    )
    response.raise_for_status()
    data = response.json()
    return [parse_volume(item) for item in data.get("items", [])]

def parse_volume(item: dict) -> dict:
    info = item.get("volumeInfo", {})
    return {
        "google_books_id": item["id"],
        "title": info.get("title", "Unknown Title"),
        "authors": info.get("authors", []),
        "description": info.get("description", ""),
        "thumbnail": info.get("imageLinks", {}).get("thumbnail", ""),
        "published_date": info.get("publishedDate", ""),
        "categories": info.get("categories", []),
    }

Caching to Stay Within Rate Limits

Making an API call on every search would quickly hit rate limits and add latency. I used Django's cache framework with a Redis backend (via Supabase) to cache search results for 24 hours.

from django.core.cache import cache

def search_books_cached(query: str) -> list[dict]:
    cache_key = f"books_search_{query.lower().replace(' ', '_')}"
    cached = cache.get(cache_key)
    if cached:
        return cached
    results = search_books(query)
    cache.set(cache_key, results, timeout=86400)  # 24 hours
    return results

Connecting Supabase as the Database

I used Supabase (PostgreSQL) as the database backend instead of SQLite. The connection is straightforward in Django — Supabase provides a standard PostgreSQL connection string that you drop into DATABASES in settings.

DATABASES = {
    "default": {
        "ENGINE": "django.db.backends.postgresql",
        "NAME": env("SUPABASE_DB_NAME"),
        "USER": env("SUPABASE_DB_USER"),
        "PASSWORD": env("SUPABASE_DB_PASSWORD"),
        "HOST": env("SUPABASE_DB_HOST"),
        "PORT": "5432",
    }
}

One thing to watch: Supabase's free tier connections can time out. I added CONN_MAX_AGE: 0 to prevent Django from trying to reuse stale connections.

The Review System

Once a book is in the local database (either from a search or a direct lookup by ID), users can write reviews. Each review is linked to both a user and a book:

class Review(models.Model):
    book = models.ForeignKey(Book, on_delete=models.CASCADE, related_name="reviews")
    user = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.CASCADE)
    rating = models.IntegerField(choices=[(i, i) for i in range(1, 6)])
    content = models.TextField()
    created_at = models.DateTimeField(auto_now_add=True)

What I'd Do Differently

If I were building NovelNet again, I'd implement a proper background task queue (Celery) for pre-fetching books instead of lazy-loading them on first search. I'd also add full-text search on the local database using PostgreSQL's built-in tsvector for faster, more relevant results on frequently-searched titles.

The live version is deployed on Render and available to try — the search works fast thanks to the caching layer.
