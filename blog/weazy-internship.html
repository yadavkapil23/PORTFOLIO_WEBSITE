What I Built at Weazy as a Web Dev Intern

Kapil 2024 6 min read

My internship at Weazy was the first time I had to write production code that real users and real business processes depended on. I went in as a third-year CSE student who had built side projects. I came out understanding what "production-ready" actually means.

The Context: What Weazy Does

Weazy is a platform in the e-commerce space. One of their core challenges was product catalog management — specifically, getting images of products from various sources, processing them, and making them available in their system. The manual workflow was slow, error-prone, and didn't scale.

My job was to automate parts of that pipeline using Python, Flask, and Azure services.

Project 1: Image Scraping Pipeline

The first thing I built was a web scraping pipeline that could pull product images from specified URLs and store them in a structured format. I used Python's requests and BeautifulSoup libraries for the scraping layer, with custom logic to handle different page structures.

import requests
from bs4 import BeautifulSoup

def scrape_images(url: str) -> list[str]:
    response = requests.get(url, timeout=10)
    soup = BeautifulSoup(response.text, "html.parser")
    images = soup.find_all("img", src=True)
    return [img["src"] for img in images if is_product_image(img)]

The tricky part was is_product_image — figuring out which images were actually product images versus icons, logos, and banners. I built a heuristic based on image dimensions and CSS classes that got us to about 85% accuracy.

Project 2: Flask API Layer

The scraping logic needed to be callable by the rest of the team's systems. I wrapped it in a Flask API with endpoints for triggering scraping jobs, checking job status, and retrieving results.

@app.route("/scrape", methods=["POST"])
def trigger_scrape():
    data = request.get_json()
    job_id = queue_scraping_job(data["url"])
    return jsonify({"job_id": job_id, "status": "queued"})

I learned a lot about API design here — specifically, that long-running tasks should never be synchronous in a web server. I moved the scraping to a background queue (using Python's threading module initially, then moved to a proper task queue) and returned a job ID immediately.

Project 3: Azure-Ready Microservices

The third major piece was packaging the services for Azure deployment. I containerized each service with Docker, wrote deployment configs, and set up a basic CI pipeline so the team could deploy updates without manual steps.

This was my first real experience with containerization in a professional context. I had used Docker before, but making it work with Azure App Service, handling environment variables properly, and setting up health checks was a different level of challenge.

The Result: 60% Reduction in Manual Work

By the end of the internship, the pipeline was processing product image batches automatically. Tasks that previously took hours of manual downloading, renaming, and uploading were reduced to a few minutes of monitoring. The 60% reduction in manual catalog work was measured by comparing average time-per-product before and after.

What I Took Away

The biggest lesson wasn't technical — it was about communication. When I hit a blocker, I had to explain it clearly to a non-technical stakeholder. That forced me to understand problems more deeply. If I can't explain what's broken in plain terms, I don't fully understand it yet.

The second lesson: always handle failures gracefully. Production systems hit edge cases constantly. Scraping a URL that returns a 429, an image that's actually a placeholder GIF, a server timeout at 3am — your code needs to handle all of it without crashing the whole pipeline.
