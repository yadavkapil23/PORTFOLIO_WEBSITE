Building a RAG Pipeline from Scratch with LangChain & FAISS

Kapil 2024 10 min read

When I started building Neobot, I wanted to understand every layer of a RAG (Retrieval-Augmented Generation) system — not just plug in a library and call it done. This is a full walkthrough of how I went from zero to a working assistant that retrieves relevant context and generates accurate answers.

What is RAG and Why Does It Matter?

Large Language Models (LLMs) are impressive, but they have a fundamental problem: their knowledge is frozen at training time. If you ask GPT-4 about something that happened last week, or about a private document in your company, it simply doesn't know.

RAG solves this by combining two things: a retrieval system that fetches relevant documents, and a language model that uses those documents to generate a grounded answer. Instead of relying purely on what the model has memorized, you feed it fresh, relevant context at inference time.

RAG is not just a pattern — it's the difference between a chatbot that hallucinates and one that actually knows what it's talking about.

The Architecture

My pipeline has four stages:

  • Document loading — ingest PDFs, text files, or web pages
  • Chunking — split documents into overlapping chunks
  • Embedding & indexing — convert chunks to vectors and store in FAISS
  • Retrieval & generation — at query time, retrieve top-k chunks and pass to the LLM

Step 1: Loading and Chunking Documents

The first challenge is getting text into a usable format. I used LangChain's document loaders to handle PDFs and plain text, then split them into chunks with overlap to preserve context across boundaries.

from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50
)
chunks = splitter.split_documents(docs)

The overlap is important — without it, a sentence split across two chunks loses its meaning. 50 characters of overlap was enough for my use case.

Step 2: Embedding and FAISS Indexing

Each chunk is converted into a vector using an embedding model. I used OpenAI's text-embedding-ada-002 for its balance of cost and quality, then stored the vectors in a FAISS index for fast similarity search.

from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import FAISS

embeddings = OpenAIEmbeddings()
vectorstore = FAISS.from_documents(chunks, embeddings)
vectorstore.save_local("faiss_index")

FAISS (Facebook AI Similarity Search) is a library built specifically for fast nearest-neighbor search over dense vectors. For a local index with thousands of documents, it's extremely fast — sub-millisecond lookups.

Step 3: Retrieval at Query Time

When a user asks a question, I embed the query and find the top 4 most similar chunks in the FAISS index.

vectorstore = FAISS.load_local("faiss_index", embeddings)
retriever = vectorstore.as_retriever(search_kwargs={"k": 4})
docs = retriever.get_relevant_documents(user_query)

Step 4: Generation with Context

The retrieved chunks are injected into a prompt template and passed to the LLM. I used LangChain's RetrievalQA chain to wire everything together cleanly.

from langchain.chains import RetrievalQA
from langchain.chat_models import ChatOpenAI

llm = ChatOpenAI(model="gpt-3.5-turbo", temperature=0)
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=retriever,
    return_source_documents=True
)
result = qa_chain({"query": user_query})

Serving It with FastAPI

I wrapped the pipeline in a FastAPI endpoint so it could be called from any frontend or tested via Postman. The whole thing runs in Docker, making it easy to deploy anywhere.

@app.post("/ask")
async def ask(query: QueryRequest):
    result = qa_chain({"query": query.question})
    return {"answer": result["result"]}

What I Learned

Building this from scratch taught me things no tutorial covers: chunk size has a massive impact on retrieval quality, embedding model choice matters more than the LLM choice for factual accuracy, and FAISS is incredibly powerful for local development before you move to a hosted vector DB like Pinecone or Weaviate.

The full project is on GitHub as Neobot_RAG. If you're building something similar, start with small chunks, test your retrieval quality independently before adding generation, and always measure hallucination rate.
