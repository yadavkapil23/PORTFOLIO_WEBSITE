My Python AI & Backend Stack Explained

Kapil 2025 5 min read

People often ask what tools I use and why. Here's a complete breakdown of my current stack — the tools I reach for when building AI systems and backend APIs — with honest notes on what I like and what I'd replace in a production context.

Python — The Foundation

Everything I build runs on Python. It's not the fastest language, but it has the best ecosystem for AI/ML work by a wide margin, and the productivity gains are real. I use Python 3.11+ for all new projects — the performance improvements over 3.9 are significant, and the newer syntax for type hints is cleaner.

FastAPI — For APIs

FastAPI is my default for building APIs. The reasons are straightforward: automatic OpenAPI documentation, native async support, and Python type hints that double as input validation via Pydantic. A basic endpoint looks like this:

from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class QueryRequest(BaseModel):
    question: str
    max_results: int = 5

@app.post("/ask")
async def ask(body: QueryRequest):
    result = await process_query(body.question, body.max_results)
    return {"answer": result}

The auto-generated docs at /docs alone save hours during development and integration. I use Django when I need a full web framework with auth, admin, and ORM built in — but for pure APIs, FastAPI wins.

LangChain — For AI Pipelines

LangChain is the glue that connects LLMs, vector stores, retrievers, and tools into coherent pipelines. It abstracts away a lot of boilerplate, but it's worth understanding what happens under the hood — especially for debugging when something goes wrong in a chain.

My honest take: LangChain is excellent for prototyping. For production, I sometimes prefer writing the orchestration logic myself because LangChain's abstractions can make debugging difficult when you're deep in a complex chain.

FAISS — For Vector Search

FAISS (Facebook AI Similarity Search) is what I use for local vector search in my RAG systems. It's extremely fast for nearest-neighbor lookups and works entirely in-memory or on disk — no external service required. For development and small to medium datasets, it's perfect.

For production at scale, I'd move to a hosted vector database — Pinecone, Weaviate, or Qdrant — but FAISS gets you very far without the operational overhead.

Docker — For Packaging

Everything I deploy is containerized. Docker ensures the environment is consistent from my laptop to CI to production. My typical Python Dockerfile:

FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

I use uvicorn (not gunicorn) for FastAPI apps since it's async-native. For Django, gunicorn is the right choice.

PostgreSQL + Supabase

PostgreSQL is my database of choice — it's reliable, has excellent JSON support, and full-text search built in. I use Supabase as a hosted PostgreSQL provider for projects that don't need a self-managed database. Supabase also provides auth and a real-time API layer, which saves setup time.

Azure — For Cloud Deployment

I've used Azure App Service for deploying containerized apps. It integrates well with GitHub Actions for CI/CD, and the free tier is generous enough for personal projects and demos. For compute-heavy AI workloads, I'd look at Azure Container Instances or AKS.

GitHub Actions — For CI/CD

Every project I deploy has a GitHub Actions workflow. At minimum: run tests on every push, build a Docker image on merge to main, deploy to the hosting platform. Automating this from day one prevents the "it works locally" class of deployment bugs entirely.

What I'd Add

The tool I'm most actively learning right now is Kubernetes for container orchestration — specifically for deploying multi-service AI systems where the retriever, generator, and API layer need to scale independently. Kubernetes is complex but the control it gives you over deployment is worth it at scale.

I'm also exploring LlamaIndex as an alternative to LangChain for RAG pipelines — it's more focused and has cleaner abstractions for document indexing specifically.
