Deploying a CLI Tool on Azure App Service with GitHub Actions

Kapil 2024 7 min read

URLZAP started as a simple Python script I wrote to shorten URLs from the command line. It used argparse, called a URL shortening API, and printed the result. Three minutes of work. The interesting part came when I decided to deploy it as a proper cloud service and automate the entire release process with GitHub Actions.

From Local Script to Web Service

The original CLI tool was purely local. To make it accessible as a web service, I wrapped it in a Flask application that exposed the same functionality over HTTP. The core logic stayed the same — only the interface changed.

@app.route("/shorten", methods=["POST"])
def shorten():
    data = request.get_json()
    original_url = data.get("url")
    if not original_url:
        return jsonify({"error": "URL required"}), 400
    short = generate_short_code(original_url)
    store_mapping(short, original_url)
    return jsonify({"short_url": f"{BASE_URL}/{short}"})

Containerizing with Docker

Before touching Azure, I containerized the app with Docker. This was non-negotiable — it guaranteed that what worked on my machine would work in the cloud without any "works on my machine" surprises.

FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 8000
CMD ["gunicorn", "app:app", "--bind", "0.0.0.0:8000"]

I used gunicorn instead of Flask's development server — an important distinction for production. Flask's built-in server is single-threaded and not designed for real traffic.

Setting Up Azure App Service

I used Azure App Service's container deployment option, which takes a Docker image and runs it. The setup through the Azure portal is straightforward: create a resource group, create an App Service plan (I used the free tier for this project), and configure the container settings.

The key Azure configuration I needed:

  • Set WEBSITES_PORT to 8000 to match the exposed port
  • Configure the startup command if needed
  • Set environment variables through the App Service configuration panel

Automating Deployment with GitHub Actions

This is the part I enjoyed most. Instead of manually pushing a new Docker image every time I made a change, I wrote a GitHub Actions workflow that does it automatically on every push to main.

name: Deploy to Azure

on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Login to Azure
        uses: azure/login@v1
        with:
          creds: ${{ secrets.AZURE_CREDENTIALS }}

      - name: Build Docker image
        run: docker build -t urlzap:latest .

      - name: Deploy to Azure App Service
        uses: azure/webapps-deploy@v2
        with:
          app-name: urlzap
          images: urlzap:latest

The AZURE_CREDENTIALS secret is a service principal JSON that gives GitHub Actions permission to deploy to my Azure subscription. You generate it with the Azure CLI:

az ad sp create-for-rbac --name "urlzap-deploy" \
  --role contributor \
  --scopes /subscriptions/{subscription-id} \
  --sdk-auth

What Broke and How I Fixed It

The deployment pipeline didn't work perfectly first try. The main issue was port binding — Azure App Service routes traffic to port 80 by default, but my container was listening on 8000. Setting the WEBSITES_PORT app setting fixed it.

The second issue was cold start latency on the free tier. The service would sleep after inactivity, causing 10-15 second delays on the first request. For a production app you'd pay for an always-on tier, but for a project it's acceptable.

The End Result

URLZAP is live on Azure App Service. Every push to the main branch triggers a fresh deployment automatically. The whole CI/CD pipeline from commit to live takes about 3 minutes. It's the kind of setup I now apply to every project I deploy.
