RAG Based Assistant

A retrieval-augmented generation chatbot that answers questions from your own documents — built with LangChain, FAISS vector search, FastAPI, Docker, and Kubernetes.

The Problem

Large language models are powerful, but they have a fundamental limitation: their knowledge is frozen at training time. They can't answer questions about your private documents, internal data, or anything that happened after their cutoff date.

The RAG Based Assistant (Neobot) solves this by combining a vector-based retrieval system with a language model — giving it the ability to answer questions grounded in real, up-to-date documents.

How It Works

The pipeline has four stages: documents are loaded and split into overlapping chunks, each chunk is converted to a vector embedding and stored in a FAISS index, at query time the most relevant chunks are retrieved, and finally the LLM generates an answer using those chunks as context.

User query → Embed query → FAISS search → Top-K chunks
→ Inject into prompt → LLM → Grounded answer

Document Processing

Documents are split using LangChain's RecursiveCharacterTextSplitter with a chunk size of 500 tokens and 50-token overlap. The overlap prevents context loss at chunk boundaries — crucial for accurate retrieval on long documents.

Vector Search with FAISS

FAISS (Facebook AI Similarity Search) stores document embeddings and performs fast nearest-neighbor lookups. For a corpus of thousands of documents, FAISS can return the top 4 most relevant chunks in under a millisecond — without any external database service.

FastAPI Backend

The retrieval and generation pipeline is served via a FastAPI application. The /ask endpoint accepts a JSON body with the user's question and returns the generated answer along with the source documents used.

Docker + Kubernetes Deployment

The application is fully containerized with Docker and orchestrated with Kubernetes — allowing the retrieval and generation services to scale independently based on load. A Kubernetes deployment manifest controls replicas, resource limits, and health checks.

Results & Learnings

The system achieves significantly lower hallucination rates compared to a vanilla LLM — because every answer is anchored to retrieved source material. The key insight from this project: chunk size and embedding model quality matter more than which LLM you use for the generation step.

Future improvements: moving from FAISS to a hosted vector DB (Pinecone or Weaviate) for persistent storage, and adding a re-ranking step to improve retrieval precision.
